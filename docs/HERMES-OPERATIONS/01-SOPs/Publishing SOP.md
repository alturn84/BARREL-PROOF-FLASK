## Purpose

This SOP governs the content publishing workflow for Barrel Proof. It defines how content moves from generation to live publication, what validation is required, and what approvals are needed.

## Standard Publishing Flow (Daily Content)

1. Cron Job 2 generates all content types
2. Content is validated by pipeline (schema, completeness, quality checks)
3. Validated content is committed and pushed to GitHub
4. Render deploys to production
5. Hermes logs the completed publication in `06-LOGS/`

No manual intervention required for standard daily content unless validation fails.

---

## Content Validation Requirements

Before any content is committed, it must pass:

**Structural validation:**
- [ ] All required fields present
- [ ] No null or placeholder values
- [ ] Correct JSON schema (for data-driven content)

**Editorial validation:**
- [ ] No unsupported statistical claims
- [ ] No fabricated quotes or invented data
- [ ] Reads at professional journalism standard
- [ ] No sensationalism or unsupported speculation
- [ ] Tone matches Barrel Proof voice (see `04-EDITORIAL/Voice & Tone Guide.md`)

**If content fails validation:**
- Do not commit
- Log the failure in `00-INBOX/`
- Alert Allan if the failure cannot be resolved by runbook

---

## Social Content Publishing Flow

1. Hermes drafts post
2. Post is staged in `10-SOCIAL/` with date and content type
3. Allan reviews and approves
4. Allan or designated publisher posts manually

Hermes **never posts directly** to X or any platform. This rule has no exceptions.

---

## Emergency Publishing

If a breaking news item requires rapid publication outside the normal pipeline:
1. Alert Allan Turner
2. Allan determines scope and timeline
3. Claude Code implements any required changes
4. Hermes generates content under Allan's direction
5. Allan approves before anything goes live
