# Runbook: Gemini API Quota Failure
**Version:** 1.0
**Last Modified:** 2025-06-06
**Applies To:** Gemini API used in Cron Job 2 content generation

---

## Symptoms

- Cron Job 2 fails or produces incomplete content
- API responses return 429 (Too Many Requests) or quota exceeded errors
- Content generation produces partial output — some sections generated, others missing
- Error logs reference quota limits, rate limits, or billing thresholds

---

## Diagnosis

### Step 1 — Confirm it is a quota issue

Check Cron Job 2 logs for:
- HTTP 429 responses
- Error messages containing: `quota`, `rate limit`, `exceeded`, `billing`

### Step 2 — Identify quota type

| Quota Type | Description |
|-----------|-------------|
| Daily quota | Total requests per day limit reached |
| Per-minute rate limit | Too many requests in a short window |
| Token limit | Individual request exceeded token limit |
| Billing quota | Account billing threshold reached |

Each requires a different response.

---

## Recovery Steps

### Scenario A — Per-minute rate limit

**Recovery Attempt 1:**
- Wait 60–90 seconds
- Retry the failed content generation request
- If successful: continue pipeline
- Log the delay

**Recovery Attempt 2:**
- If rate limit error persists after waiting: reduce concurrent requests if applicable
- Retry individual content generation sequentially rather than in parallel
- Log the outcome

### Scenario B — Daily quota exhausted

This cannot be resolved by Hermes without account intervention.

1. Document which content types were generated successfully before quota was reached
2. Document which content types are missing
3. Alert Allan immediately — this requires account or billing action
4. Do not attempt to generate placeholder or fabricated content to fill gaps
5. Publish only what was successfully generated; leave missing sections absent rather than filling with bad content

### Scenario C — Individual request token limit

**Recovery Attempt 1:**
- Identify which prompt hit the token limit
- If the prompt can be reduced (trim context, reduce data payload): do so
- Retry the request
- Log the adjustment

**Recovery Attempt 2:**
- If token limit cannot be resolved by prompt reduction: escalate to Allan
- Document which content type failed and why

### Scenario D — Billing quota

Escalate to Allan immediately. This requires account action.

---

## Hard Rules

- **Never generate placeholder or invented content to fill gaps caused by API failure**
- **Never post incomplete content without Allan's knowledge**
- **If content is missing due to API failure: Allan decides whether to publish partial content or delay**

---

## Escalation Criteria

Escalate to Allan if:
- Daily quota is exhausted
- Billing threshold is reached
- Recovery attempts fail and content is missing
- Any scenario where content cannot be generated by 6:30 AM ET

---

## Post-Recovery

- [ ] Confirm all content types are generated (or document which are missing)
- [ ] Log incident in `06-LOGS/`
- [ ] Note API error type and time for pattern tracking
- [ ] If partial publication: confirm Allan approved
